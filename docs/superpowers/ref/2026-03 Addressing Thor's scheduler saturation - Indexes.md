# Background

[Part 1](?tab=t.0) "largely ignored" the task load generated by metastore queries since [other efforts](https://docs.google.com/document/d/1VVij7XQ9chHcIwCSo2BSgh7oK1hHjIH7XYM-znaGURo/edit?tab=t.mrbdhoktcksm#heading=h.s93vj29x4wud) are ongoing to temporarily address the metastore's load, but it's still true that metastore tasks generate a lot of load on the scheduler.

To understand the problem, let's review how a metastore lookup works today:

* Open one TOC per 12h window being queried.  
  * For each TOC, find all index objects that have *any* data overlapping with the queried time range.  
    * For each index object, scan through the streams section to find stream IDs matching your LogQL stream selector.  
    * Find postings for those stream IDs in the pointers section.  
    * Additionally: if your query filters on structured metadata, intersect these with the postings for bloom filters on those columns.  
  * Union all index object results together (across all TOCs) to form the final list of data objects to open. 

Two parts of this flow cause inefficiencies at ops scale:

* We have a *lot* of index objects, thanks to our write volume and number of Kafka partitions. To make logs from flushed data objects available for querying quickly, we flush index objects after they accumulate references for [16](https://github.com/grafana/kube-manifests/blob/eca48815f42fb3c597b3b2924eeea1953d86cef6/flux/ops-eu-south-0/loki-ops-002/StatefulSet-dataobj-index-builder.yaml#L36) data objects.

* By only looking at time ranges, TOCs typically resolve to most index objects when querying for a wide time range, such as the incredibly common "Last 24h" query.

This tab explores what storage locality should look like for indexes.

# Index storage locality  

Index objects are affected by storage locality in a very similar way to log objects:

* **Data locality** means minimizing the number of sections *where postings for queried predicates* (columns) are stored.   
* **Time locality** means minimizing the number of sections where *timestamps* for queried predicates (columns) are stored. 

However, unlike log object storage locality, there is less dimensionality to play with here, and the locality is less impacted by query patterns. 

# One sort schema for indexes 

An index object technically holds two types of indexes:

* A bloom filter index for structured metadata columns, and   
* an inverted index for label values.

Bloom filter indices typically take up the majority of the space in an index object section. Since querying structured metadata is quite rare, it makes sense to store these indexes far away from the label value index to minimize impact. This means that users that never query for structured metadata columns don't have to "pay the cost" of the large bloom filter index causing the queried predicates being stored far apart. Given this, the query schema will start with the index type.

Below the index type, we can store postings for one label at a time, sorted naturally. Each posting in the pointers section can continue to be in timestamp order. This gives our indexes the sort schema of \[\_\_index\_type\_\_, \_\_label\_\_, \_\_timestamp\_\_\][^1]

**NOTE:** While not immediately obvious, this sort schema requires the metastore to be aware that postings can spill across multiple sections, especially once compaction is introduced. This is different from how the metastore is structured today, where each log object can only be referenced by exactly one index object. The metastore query code will need to ensure it forms the final list of log objects by considering both the full predicate list and the set of columns indexed in each index object section. 

# New index object layout

The existing index object layout, with its streams and pointers sections, does not easily support this new sort schema: stream IDs are compound keys, but we want the sort order to be one key at a time, making the layout incompatible with the design. 

One hacky fix for this could be to force each stream ID in the streams section to only have one label at a time:

| ID | Min TS | Max TS | service\_name | cluster | … | Rows | Size |
| :---- | :---- | :---- | :---- | :---- | :---- | :---- | :---- |
| 1 |  |  | a |  |  | 10 | 123 |
| 2 |  |  | b |  |  | 15 | 456 |
| 3 |  |  | c |  |  | 11 | 321 |
| 4 |  |  | d |  |  | 20 | 222 |
| 5 |  |  |  | dev |  | 10 | 123 |
| 6 |  |  |  | ops |  | 15 | 456 |
| 7 |  |  |  | prod |  | 11 | 321 |
| 8 |  |  |  | prod-2 |  | 20 | 222 |

Here, each unique key-value pair receives its own stream ID, and no row has more than one label set. Then, mappings to streams in log objects would receive one posting per label: if an indexed stream 13 was {service\_name="a", cluster="ops"}, it would have two postings for ID 1 \-\> 13 and ID 6 \-\> 13\. 

While this approach works, it's quite a hack around how the streams section was intended to be used. Many columns would have many nulls, which aren't free to store or decode. 

Instead, I propose we change index objects to use two new fully independent sections: a **stats section** and a **postings section.** 

## Stats section

The **stats section** contains high-level information about individual sort schema keys in log objects. The stats section is primarily intended to be used by the [compactor](?tab=t.kbzjr4jhrrq3), and will not be used for query planning.  

The stats section has 9 column types:

* **Object path**: The path to the logs object of the section being indexed.  
* **Section index**: The indexed section number within the logs object.   
* **Run ID:** Which "run" this section belongs to. (See [part 4](?tab=t.kbzjr4jhrrq3) for more information).   
* **Sort schema**: A comma-delimited list of column names which defines the sort schema for this key.   
* **Label**: A named label column for one of the elements of the sort schema. There is one label column per sort schema key.   
* **Min timestamp:** The minimum timestamp across all records in this sort key.  
* **Max timestamp**: The maximum timestamp across all records in the sort key.   
* **Row count:** The number of records in the sort key.   
* **Uncompressed size:** The total number of bytes for all records in the sort key. 

This is very similar to the existing streams section used by index objects. The two primary differences are only having one label per element of the sort key (minus timestamp), and having a column which describes the sort schema (which allows to detect two objects having different schemas). 

| Path | Index | Run ID | Schema | service\_name | … Other columns |
| :---- | :---- | :---- | :---- | :---- | :---- |
| obj-a | 1 | ABC | service\_name | bar | … |
| obj-a | 1 | ABC | service\_name | foo | … |

To keep stats sections mergeable, we will define some kind of sort schema for them. It's not yet known what locality we need out of these, so it's not clear what kind of sort schema they should have. We can start with sorting these by their defined sort schemas (the set of Label columns), matching the sort schema of log objects. If two objects have different sort schemas, we can tiebreak by sorting by the literal comma-delimited SortSchema column first. 

## Postings section

The **postings section** is the equivalent to the pointers section, but is "flattened" to permit following the index object sort schema. It has 10 column types:

* **Kind:** The kind of posting, either Bloom (for bloom filters) or Label (for label values).  
* **Object path:** The path to the logs object of the section being indexed.  
* **Section index:** The indexed section number within the logs object.  
* **Column name:** A column containing the indexed column name.   
* **Label value:** The indexed label value. Only set for Kind=Label.   
* **Bloom filter**: The indexed bloom filter. Only set for Kind=Bloom.   
* **Stream ID bitmap:** Which stream IDs in the indexed section have any entry for this row. For Kind=Label, this stream IDs that have the exact key-value pair. For Kind=Bloom, this reports the stream IDs that contributed to the bloom filter for the corresponding column name.  
* **Uncompressed size:** The total number of bytes for all records with this label. Filtered by value presence when Kind=Label.   
* **Min timestamp:** The minimum timestamp across all records with this label. Filtered by value presence when Kind=Label**.**   
* **Max timestamp:** The maximum timestamp across all records with this label. Filtered by value presence when Kind=Label.

The stream ID bitmap is required to merge postings *within* a section. Without this, you can only query for individual sections that match all predicates, but that section might not have any streams that match the predicates.

Flattening the key-value entries is what permits index objects to be sortable and splittable across multiple sections:

| Kind | Path | Index | Name | Value | Bloom | Bitmap | … |
| :---- | :---- | :---- | :---- | :---- | :---- | :---- | :---- |
| Bloom | obj-a | 1 | md-1 |  | \<bloom\> | 111 |  |
| Bloom | obj-a | 1 | md-2 |  | \<bloom\> | 111 |  |
| Label | obj-a | 1 | job | distributor |  | 110 |  |
| Label | obj-a | 1 | job | worker |  | 001 |  |
| Label | obj-a | 1 | namespace | dev |  | 110 |  |
| Label | obj-a | 1 | namespace | dev-gf |  | 001 |  |
| Label | obj-a | 1 | tenant | 29 |  | 100 |  |
| Label | obj-a | 1 | tenant | 2914 |  | 010 |  |

With this in mind, the applied sort schema is \[Kind, ColumnName, ColumnValue, \_\_timestamp\_\_\].

This design comes with some known cost: paths, column names, and values will be repeated many times across the section. While compression will help with this, it doesn't help with how sections are cut based on uncompressed sizes. In the short term, it's expected that the gains in storage locality will outweigh the gains in redundant data. Longer term, this can be addressed with dictionary encoding, where you only count each unique dictionary key once and each dictionary value (an integer) towards the uncompressed size limit.  

# Updates to the TOC 

The TOC should be updated to become aware of the minimum and maximum column names indexed in each index object section. 

This will allow the TOC to prune out index objects more effectively based on query selectors, rather than looking at the queried time range alone. While indexing the full set of column names would be more effective when querying non-existent data, storing only min/max is helpful to not need to worry about cardinality, especially since there can only be one TOC object per window.

# Comparison to today 

With the new TOC and sort schema, a query for {wave="dev", container="query-engine-worker"} now looks like:

* Open one TOC per 12h window being queried.  
  * For each TOC, find all index objects that have the wave OR container label indexed *and* have data overlapping with the queried time range.  
    * For each index object, scan through the postings section to find stream IDs matching the queried predicates.  
  * Merge all index object results together (across all TOCs) to form the final list of data objects to open, with consideration to the final predicates in each index object. 

Compared to the original flow, there are two changes:

* TOC now additionally prunes based on indexed labels.   
* Merging the postings requires slightly more sophisticated logic to handle two sections having different subsets of the full queried columns.

Like with log object locality, this limits the maximum load generated by a metastore lookup to be scoped to the number of columns being queried (which is typically pretty low). 

However, as the metastore exists now, this will have limited performance gains (though pruning on indexed labels will help). The true benefits will come from reducing the total number of index objects, likely through a compaction process.

# Future changes 

In the future, dataobj index builders can potentially be removed in favour of having that data be written directly by dataobj-consumers. This would remove latency and make logs queriable faster, but shift more responsibility onto compaction for reducing the number of index objects. 

[^1]:  Ties can be broken using object name and section index, but I'm leaving these out of the sort schema for simplicity.
