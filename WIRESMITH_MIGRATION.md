# Loki → wiresmith migration — status, issues and incompatibilities

Assessment of generating Loki's protobuf Go code with
[wiresmith](https://github.com/grafana/wiresmith) instead of the current
`protoc --gogoslick_out` toolchain, **updated 2026-06-10** against wiresmith
`v0.0.0-20260609114539-bee7aa4`. This supersedes the previous assessment: two
of its four TL;DR blockers are resolved (gRPC codegen now exists; the
annotation surface now covers stdduration/casttype/message-customtype), one is
narrowed but still real (go_package collisions), and one new hard constraint
was discovered (no cross-runtime message embedding).

A partial migration has been **executed on this branch** (`wiresmith`); see
[Migration status](#migration-status).

---

## TL;DR

- **8 of 38 protos migrated, building, and passing tests**: the four engine
  protos, `compactionv2`, both `stats` protos, and `limits` (the first proto
  with gRPC services).
- **The rest of the proto suite is one import-connected cluster rooted at
  `logproto`/`push`**, and that cluster is hard-blocked by wiresmith's
  one-go_package-per-proto-package rule: `pkg/logproto/logproto.proto`
  *imports* `pkg/push/push.proto` and both declare `package logproto` with
  different `go_package`s, so they cannot even share a compile walk. `-M`
  does **not** bypass the check (verified — see W1).
- A second structural constraint compounds this: **wiresmith-generated code
  cannot embed message types generated by another runtime** (it calls
  `UnmarshalWithDepth`/`Compare`/`Equal`-with-wiresmith-semantics on
  sub-message fields — see W2). Migration must therefore proceed in
  import-topological order, and every proto a migrated file imports must be
  migrated in the same step (or bridged via a customtype adapter).
- Everything else found in the old assessment is now either supported
  (stdtime, stdduration, casttype, jsontag, customname, message customtype,
  gRPC incl. streaming) or mechanical (value-type call-site adaptation,
  enum-prefix renames, GoString shims).

Toolchain integration:

- `make wiresmith-protos` (wired into `make protos`) stages each *generation
  group* into a disjoint temp tree mirroring the repo layout and runs one
  `wiresmith --proto_path=<stage> --out=. --module=github.com/grafana/loki/v3`
  invocation per group. Groups exist because of the go_package rule (W1):
  both stats protos declare `package stats`, and `limits` shares
  `package proto` with `compactionv2`.
- `clean-wiresmith-protos` removes the `_compare`/`_equal`/`_reflect`/`_grpc`
  sibling files; migrated protos are filtered out of `PROTO_DEFS` so the gogo
  pipeline no longer touches them.
- `go.mod` requires `github.com/grafana/wiresmith` with a local `replace`
  (`/Users/oleg-kozlyuk/Projects/wiresmith/wiresmith`); `protohelpers` is
  vendored. Before merging anything like this for real, wiresmith needs a
  published tag.

   imports them (`queryrange`, `scheduler`, `frontend v1/v2`, `ruler`,

   `protohelpers`) to that module's dependency graph. That is probably
   acceptable (it's a small dependency, similar to gogo today) but is a
   product decision; alternatively wiresmith could be taught to emit
   self-contained helper code.

## Wiresmith blockers (ranked)

### W1 — one `go_package` per proto package across the whole walk; `-M` does not bypass it

- **Proto/field:** `pkg/push/push.proto` + `pkg/logproto/*.proto` (all
  `package logproto`); also `package proto` ×3 dirs and `package stats` ×2
  dirs (worked around via generation groups).
- **Gogo behavior:** protoc compiles each file independently;
  `paths=source_relative` lets one proto package span Go packages.
- **Failing experiment:** stage `push.proto` + `logproto.proto` (+ imports)
  in one tree:
  ```
  error: inconsistent go_package for package "logproto":
  pkg/logproto/logproto.proto declares "github.com/grafana/loki/v3/pkg/logproto"
  but pkg/push/push.proto declares "github.com/grafana/loki/pkg/push"
  ```
  `-M a/stats.proto=…
  -M b/stats.proto=…` on a minimal two-file repro fails identically: the
  check (`collectGoPackages`, `compiler/generator/generator.go:595`) reads
  each file's literal `option go_package`, not the `-M`-resolved destination.
  Positional scoping doesn't help either — the check runs over the full
  compile walk, not just emitted files.
- **Workaround:** disjoint staging trees per "generation group" — works only
  when the colliding files don't import each other. `logproto` imports
  `push`, so **no workaround exists** for the main cluster short of renaming
  `package logproto` on the wire (an audit-heavy breaking change for
  reflection/`Any`/FQN consumers).
- **Suggested wiresmith feature:** honor `-M` overrides inside
  `collectGoPackages` (protoc-M semantics), or relax the rule to
  per-directory consistency. The error message's own framing suggests `-M`
  was meant to be the escape hatch.
- **Severity:** **blocker** — gates ~28 of the remaining 30 protos.

### W2 — generated code can't reference messages generated by another runtime

- **Proto/field:** any message field whose type lives in a non-wiresmith
  package. Concretely hit: `wirepb.TaskAssignMessage.metadata` →
  `httpgrpc.Header` (gogo, vendored dskit);
  `physicalpb.IndexMerge.runs` → `proto.RunRef` (was gogo, forced
  `compactionv2.proto` into the same migration batch).
- **Failing experiment:** import a non-emitted proto and emit a file
  referencing its message: the generated `.pb.go` calls
  `m.Metadata[i].UnmarshalWithDepth(...)`, `_compare.pb.go` calls
  `.Compare(...)` (value arg) and `_equal.pb.go` `.Equal(...)` — gogo types
  have no `UnmarshalWithDepth`, so the package does not compile.
- **Workarounds (both used on this branch):** (a) migrate the imported proto
  in the same batch; (b) declare a wire-identical local message and bridge
  the Go type with a `customtype` adapter (`wirepb.Header` +
  `HeaderAdapter` delegating to the gogo `httpgrpc.Header` methods).
- **Suggested wiresmith feature:** a documented interop story for
  sub-messages from other runtimes — e.g. fall back to
  `Size/MarshalToSizedBuffer/Unmarshal` (the gogo-compatible subset wiresmith
  already emits) when the imported type's package is not wiresmith-generated,
  or an option to suppress Compare/Equal emission per field.
- **Severity:** **major** — defines migration order (leaf-first), makes
  incremental migration of tightly-coupled protos all-or-nothing per cluster.
  Note the *reverse* direction works: gogo-generated code embedding
  wiresmith types compiles and round-trips (verified by
  `queryrange`/`logproto`/frontend tests) as long as `GoString` shims are
  provided (see m3).

### W3 — unused Go import emitted when customtype replaces the only cross-file reference

- **Proto/field:** `wirepb.TaskAssignMessage.metadata` while it was declared
  `repeated httpgrpc.Header … [(wiresmith.options.customtype) = "HeaderAdapter"]`.
- **Failing experiment:** generation succeeds but the emitted `wirepb.pb.go`
  contains `import "github.com/grafana/dskit/httpgrpc"` with zero uses →
  `imported and not used` compile error.
- **Workaround:** declare a local message for the field's wire type so no
  foreign import exists (done).
- **Suggested wiresmith feature:** drop imports that end up unreferenced
  after customtype substitution.
- **Severity:** minor (workaround is easy, error is loud).

### W4 — presence bitmap only set by Unmarshal; getters and equality observe it

- **Field:** every singular message field without `pointer = true`, e.g.
  `frontendv2pb.QueryResultRequest.stats` (gogo parent, wiresmith child).
- **Failing experiment:** `pkg/lokifrontend/frontend/v2`
  `TestProtobufBackwardsCompatibility` — `require.EqualValues` between a
  hand-constructed literal and a wire-decoded value fails on the unexported
  `fieldsPresent [1]uint64` (decoded = 3, literal = 0). Also:
  `Get<MessageField>()` returns `nil` for hand-constructed values whose
  presence bit is unset even when the field content is non-zero.
- **Workaround:** compare via the generated `Equal` (ignores the bitmap);
  avoid `reflect.DeepEqual`/`cmp.Diff` across construction provenance.
- **Suggested wiresmith feature:** documentation at minimum; possibly
  `Set<F>()` helpers or content-based `Get` for value-typed message fields.
- **Severity:** minor-to-moderate — a quiet behavioral trap in test code and
  any `DeepEqual`-based production logic.

## Resolved since the previous assessment

### B1 (old) — gRPC service codegen: **RESOLVED**

`X_grpc.pb.go` is now emitted automatically for service-bearing protos
(vendored protoc-gen-go-grpc; unary + server/client/bidi streaming all
verified on a synthetic proto; unary verified end-to-end via
`pkg/limits` tests). Differences from gogo's grpc plugin:

- Server implementations must embed `Unimplemented<Svc>Server`
  (`mustEmbedUnimplemented…` forward-compat contract). One-line change per
  implementation (`pkg/limits/service.go`, `pkg/limits/frontend/frontend.go`).
- Clients take `grpc.ClientConnInterface` (compatible with existing call
  sites).

### B2 (old) — partially resolved → narrowed to W1

The stats/`package proto` collisions are workable via per-group staging
trees. The logproto/push collision is not (files import each other). `-M`
exists but does not feed the consistency check.

### M1 (old) — customtype: **largely resolved**

Now covers singular **and repeated** `bytes`/`string`/**message** fields via
the `SizeWiresmith`/`MarshalWiresmith`/`UnmarshalWiresmith`/`EqualWiresmith`/
`CompareWiresmith` contract. The ULID-on-message case (engine protos) is
migrated with ~50 lines of adapter methods delegating to the existing gogo
implementations (`pkg/engine/internal/proto/ulid/ulid.go`). Scalar
customtype (gogo's `model.Time` on `int64 from/through`) is still not
supported as *customtype*, but `casttype` covers it (verified by compile +
round-trip experiment: `type Time int64` over an `int64` field) since gogo's
`customtype+nullable=false` on int64 produced the same Go shape.

### M2 (old) — stdtime/stdduration: **RESOLVED**

Both exist, singular-only, value-typed (`time.Time`/`time.Duration`), which
matches every Loki usage (all are `nullable=false` singular). Used in
`physicalpb` (4× stdduration was the old blocker), `wirepb`,
`querier/stats`. Caveats: zero value means "unset" (no `Has` accessor) and
decoded times are UTC-normalized — no Loki breakage observed.

### M4 (old) — casttype: **RESOLVED**

`(wiresmith.options.casttype)` exists for integer scalars, sint/fixed, bool,
string, bytes. Verified `model.Time` (int64), `model.Fingerprint` (uint64)
and same-package `DetectedFieldType` (string) compile and round-trip. Not
yet used on this branch (the 5 Loki usages live in the blocked logproto
cluster). Not supported: float/enum/message/map/oneof/repeated/optional.

### M5/M6 (old) — jsontag / customname: **RESOLVED**

`(wiresmith.options.jsontag)` is a verbatim drop-in (no `,omitempty`
appended) — 84 tags in `logqlmodel/stats` migrated mechanically and the
JSON output of `stats.Result` is **byte-identical** to gogo (golden test
checked in). `customname` exists (not yet needed on this branch).

### M7 (old) — marshaler_all etc.: confirmed no-op

Wiresmith always emits the gogo-compatible
`Marshal/MarshalTo/MarshalToSizedBuffer/Unmarshal/Size` set; the file-level
options were deleted along with the `gogoproto` import (they would otherwise
fail to parse once `gogo.proto` is no longer imported).

## Major incompatibilities (work, but mechanical — all exercised on this branch)

### M-a — value-type field shapes (gogo `nullable` inversion)

Wiresmith defaults to `T`/`[]T` where gogo defaulted to `*T`/`[]*T`. Two
adaptation modes, both used:

- **Value adaptation** (preferred): `compactionv2` `RunRef`/`SectionRef`
  slices through `calculator/plan/index_merge/log_merge/coordinator`;
  expressionpb/physicalpb conversion helpers; `wirepb` codec. Diffs are
  small but touch many lines; the compiler finds every site.
- **`pointer = true`** where nil-ness is load-bearing or pointer shapes
  pervade an internal API: `physicalpb.{AggregateRange,AggregateVector}.grouping`
  (nil → "empty grouping" error), `PointersScan.selector`,
  `wirepb.{NackFrame,TaskStatus}.error`, `TaskStatus.contributing_time_range`,
  `TaskAssignMessage.task`, and all repeated fields in `limits` (the
  store/ring internals are pointer-based throughout).

Maps of message type are value-typed (`map[string]StreamList`) with no
pointer option for map values — `wirepb` codec adapted.

### M-b — enum constants are always prefixed

Gogoslick emitted `UNARY_OP_NOT`; wiresmith emits `UnaryOp_UNARY_OP_NOT`
with no `goproto_enum_prefix` toggle (deliberate). Mechanical rename in 5
files (expressionpb/physicalpb helpers, scheduler codec).

### M-c — oneof shape changes

Wrapper structs hold **values** (`Expression_Unary{Unary UnaryExpression}`),
the variant interface is exported (`Expression_Kind` vs gogo's unexported
`isExpression_Kind`), and `Get<Variant>()` still returns `*T` (pointer into
the wrapper), which kept `n.GetDataObject() != nil` call sites compiling
unchanged. Construction sites (`&Node_Filter{Filter: &Filter{…}}` →
`&Node_Filter{}` + in-place unmarshal, or value literal) were the bulk of
the expressionpb/physicalpb churn.

## Minor incompatibilities

### m1 — single `--proto_path`, imports resolved repo-relative

No multi-`-I` support, no sibling-dir resolution, and pointing
`--proto_path` at the repo root is impossible (the walk would parse
`vendor/**` and trip W1 plus duplicate-symbol errors on
`vendor/.../push/push.proto`). The staging-tree pattern in
`make wiresmith-protos` is the workaround; import-only dependencies (e.g.
proto2 `gogoproto/gogo.proto` for a vendored import) parse fine when staged.
The old m1 case (`resultscache/test_types.proto` importing sibling
`"types.proto"`) still needs an import-path rewrite when that cluster
migrates.

### m2 — module path / distribution: **improved**

Module path is now `github.com/grafana/wiresmith` and generated code imports
`github.com/grafana/wiresmith/protohelpers` (stable path). Still
unpublished — consumers need a `replace` until a version is tagged.
`wiresmith/options.proto` is embedded in the compiler (CLI mode), so nothing
needs vendoring on the proto side.

### m3 — no `GoString`; gogoslick callers need shims

Gogoslick-generated code calls `GoString()` on embedded message values.
Wiresmith doesn't emit it. Hand-written 6-line shims per affected type
(`pkg/logqlmodel/stats/gostring.go`: `Result`, `Ingester`, `Index`). Only
needed for wiresmith types embedded in still-gogo protos.

### m4 — `google.protobuf.Any` / WKT consumers

Unchanged from old assessment: wire-compatible structs are generated if the
`.proto` is on the path, but reflection-based `Any` packing/unpacking won't
work. Affects `resultscache/types.proto` and `rulespb` — both in the
blocked cluster anyway.

### m5 — duplicate symbols masked by gogo's per-file compilation (Loki bugs)

`service PusherRF1` defined in both `pkg/push/push-rf1.proto` and
`pkg/logproto/ingester-rf1.proto` (same `package logproto`), and the
vendored copy of `push.proto`. Protoc tolerates these because each file
compiles in isolation; wiresmith's whole-walk compile rejects them
(`symbol "logproto.PusherRF1" already defined`). Must be fixed in Loki when
the cluster migrates: rename one service or drop the duplicate file from
the walk; staging trees already exclude vendor copies.

### m6 — gogo↔protobuf-go registry split

Replaced files no longer register with gogo's registry; `_reflect.pb.go`
registers with `google.golang.org/protobuf`'s. No Loki code was found
calling gogo `proto.MessageName`/`RegisterType` on the migrated types, and
gogo `proto.Marshal/Unmarshal` still work via the `Marshaler`/`Unmarshaler`
fast paths (exercised by the scheduler wire codec tests). Worth re-checking
per package as migration proceeds (`jsonpb` in particular would break).

## Cosmetic differences

- Four (five with services) generated files per proto instead of one —
  `clean` targets updated; `*_reflect.pb.go` is large (~64KB+ descriptor
  blobs) but keeps the hot `.pb.go` lean.
- `fieldsPresent [1]uint64` per message (see W4 for the behavioral edge).
- Struct sizes/layouts differ (value fields inline sub-structs); heap
  profiles will shift, generally favorably.
- `String()` output format differs from gogoslick's (e.g. log lines that
  print whole messages change shape).
- Wire format note: gogo marshals `nullable=false` sub-messages
  unconditionally (tag + recursive empty payload); wiresmith omits empty
  value-typed sub-messages unless their presence bit is set. Decoders see
  the same logical value; byte-golden tests (like the frontend k173.bin
  fixture) still pass because decode semantics are identical.

## Recommended path forward

1. **Land the leaf-proto migrations** (this branch + the remaining isolated
   protos from "Remaining work" item 1). They capture the perf win on the
   hot engine wire path with no external-compat exposure.
2. **File the W1 `-M` issue against wiresmith** — it is the single gate for
   the rest of Loki. Honoring `-M` in `collectGoPackages` looks like a
   small, philosophy-compatible change (protoc parity).
3. **File W2/W3 issues** (cross-runtime sub-message interop; unused-import
   after customtype). W2 matters for any consumer that vendors gogo protos
   (dskit `httpgrpc`, thanos store-gateway protos).
4. After W1: migrate `push`+`logproto` together (one group), then fan out to
   the service protos cluster by cluster. Budget for the `pkg/push` module
   dependency decision and the m5 duplicate-symbol cleanups.

**Do not** rename proto packages on the wire to dodge W1 — the audit cost
(reflection, `Any`, external clients) exceeds the cost of the wiresmith
fix.
